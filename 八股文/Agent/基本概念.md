Agent就是给普通的llm一个提示和一些操作的权限和一些流程
**Agent = LLM + 工具 + 记忆 + 规划 + 执行循环**

#### Tool
Tool 是 Agent 调用外部能力的方式。比如：

- 搜索（联网查资料）
- 计算（数学运算）
- 代码执行（写代码并运行）
- API 调用（查天气、发邮件）

不同系统中的Agent，根本上是**Tool的差异化**。比如医保系统，agent会帮你拉取所有的相关政策，比如飞书，agent会帮你记录日历，提示等等。一般想要引入agent，你只要写一下你这个系统开放的Tool，然后直接套用agent模板就可快速引入了

#### MCP
MCP是支持被agent调用的外部系统提供的Tools，通过MCP提供了一个统一的调用方法

#### Skill
Skill是对Prompt调用Tools的进一步封装，Tools仅仅只有原子操作，比如查询天气、写入文件，那查询天气后写入这个天气到文件就是Skill。同时Skill支持保存和传递

#### RAG
是一类Tool，类似于关键词检索，但不是基于字的相同而是语义的相同。比如你在干什么，关键词搜索干什么的时候才能搜到，但是RAG搜做啥子也能搜到

设计要点
- 数据预处理：将文档切块（Chunking），选择合适的 Embedding 模型，存入向量数据库（如 Qdrant、Milvus）。

检索策略：
- 混合检索：结合关键词搜索（BM25）和向量搜索，提高召回率。
- 重排序（Rerank）：对检索结果进行二次排序，把最相关的信息排在前面TopK，避免上下文被无关信息“污染”。
- 注入上下文：将检索到的内容动态拼接到 Prompt 中，格式要清晰，便于模型区分“已知信息”和“检索到的信息”。

#### Prompt Engineering
是LLM二次微调的核心，能让它变得更适合完成某类任务

- 角色设定：明确 Agent 的身份、能力和边界（如“你是一个数据分析师，只能执行 SQL 查询，不能删除数据”）。
- 工具描述：清晰告诉模型每个工具是做什么的，什么时候用，参数是什么。建议使用 OpenAI Function Calling 或 JSON Mode 来定义工具，比纯文本描述更稳定。
- 思维链（CoT）：引导模型“一步一步想”，尤其是在处理复杂逻辑或多步任务时。
- 自我纠错：在 Prompt 中植入“如果工具调用失败，尝试备选方案”或“如果结果不相关，重试检索”的指令。
- 行为控制：实现Plan-Execute模式和ReAct模式

#### Memory
| 类型       | 作用                  | 实现方式          |
| -------- | ------------------- | ------------- |
| **短期记忆** | 记住当前会话的上下文（最近几轮对话）  | 会话窗口、Redis 缓存 |
| **长期记忆** | 跨会话的用户偏好、历史事实       | 向量数据库 + RAG   |
| **工作记忆** | 当前任务执行过程中的中间状态      | 状态机、上下文变量     |
| **情景记忆** | 记录关键事件（成功/失败案例）用于复盘 | 日志 + 离线分析     |
#### 多Agent协作

创建多个智能体，它们在一些细节方面不同，如Model、Memory、Prompt。主要包括几种协作方式：
- 主控agent+执行agent：类似Reactor模式，主控的负责拆解任务，给执行agent分配任务等，执行agent收到任务并返回结果
- 轮询协作：像辩论一样，每个agent有不同的性格，对同一个问题有不同的角度。那么它们之间不断辩论，类似于深度思考，最后得出一个完美的答案
- 图式编排：一个任务被定义成DAG，每个节点是一个agent，边是数据流
